# Audit Checklist

按以下顺序执行，不要跳步。

## 1. Project scaffold and naming

检查：
- 目录是否符合项目约定（如 `01_data/`, `02_code/`, `03_tables/`, `04_figures/`, `07_paper/`, `09_backup/`）
- 脚本命名是否规范（如 `NN_description.R`、`FigN_xxx.*`、`TableN_xxx.*`）
- 根目录是否存在临时文件、过期导出、重复版本
- 日志文件是否存在并持续更新：`SESSION_LOG.md`、`DECISIONS.md`

红旗：
- `test.R`, `final.R`, `temp.*`, `new new final.*`
- 图表/表格命名与正文引用不一致
- 同一结果存在多个“最终版”文件

## 2. Data provenance and extraction chain

检查：
- 原始数据是否只读保存
- 派生数据是否能追溯到来源文件
- 缺失值是否被标注为 NR/NA/不适用，而不是空白混用
- 提取表中的关键字段是否与提取报告/原文一致
- source map、提取报告、主提取表是否互相能对应

红旗：
- 结果表里有值，但无法追溯到数据或提取报告
- 提取报告有明文，主表却空缺
- 真缺失和漏提混在一起，没有说明

## 3. Code audit and reproducibility

检查：
- 主脚本是否可运行
- 依赖脚本是否顺序清楚
- 是否存在明显编程错误、对象名错、导出覆盖错、尺度错
- 是否生成预期的表格、图片、数据文件
- 是否存在旧脚本和新脚本并行输出矛盾结果

红旗：
- 代码未运行，仅靠静态编写
- 修改了结果逻辑但未同步更新输出
- 表格里 `I2`、`OR/RR/HR`、单位或百分比尺度错误

## 4. Result consistency

逐项核对：
- 表格 vs 图片
- 表格 vs 摘要
- 表格 vs 正文结果段
- `0_result_summaries.md` vs 终稿 vs 分享包
- 主分析 vs 敏感性分析 vs 附录

红旗：
- 同一指标在不同文件数值不一致
- 正文说“显著”，结果表 CI 却跨 1
- 预测区间跨无效线，但结论写成确定性保护/风险

## 5. Scientific validity

检查：
- 研究设计与分析方法是否匹配
- 因果语言是否超出研究设计支持范围
- 单篇综述/单篇Meta是否被写成强结论
- 叙述性证据是否被误当作定量证据
- 质量评价、重叠风险、异质性、偏倚是否真正进入解释

红旗：
- 观察性结果写成因果结论
- 高重叠风险/高异质性结果写成稳定结论
- 证据等级是“研究组推断”，正文却写成“源研究报告”

## 6. Writing and claim calibration

检查：
- 结论是否与证据强度匹配
- 是否用“提示/支持/方向一致/需审慎”替代过强措辞
- 图意说明是否先描述图展示什么，再解释结果，再给出结论
- 限制是否写在结论附近，而不是只埋在附录

## 7. Delivery and handoff hygiene

检查：
- 分享包是否包含最新版本结果
- README 是否能让人独立理解项目状态
- 关键输出是否都已同步更新
- 旧版副本是否仍在误导使用者

最终判定：
- **pass**：无关键错误，结论与证据匹配
- **pass with concerns**：数值正确，但存在重要边界或中等风险
- **fail**：存在结果错误、可复现性问题或明显过度结论
